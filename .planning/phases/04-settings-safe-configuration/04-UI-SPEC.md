## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline label gaps, change badges, helper text separation |
| sm | 8px | Button groups, field-to-field spacing inside a control row |
| md | 16px | Default control spacing, card internals, status stacks |
| lg | 24px | Section card padding, inter-card gaps, panel gutters |
| xl | 32px | Panel header spacing, review callout breathing room |
| 2xl | 48px | Mobile page padding top/bottom, major subsection separation |
| 3xl | 64px | Desktop page padding and top-level shell spacing |

Exceptions: 12px radius on cards and controls; 44px minimum height for every settings input and action button; 56px minimum height only for any large stacked action button reused from existing control surfaces.

Source: existing shell/card spacing in `web/src/styles.css`; 44px touch-target default chosen as the phase-standard exception.

## Color

|------|-------|-------|
| Dominant (60%) | #12161b | App shell background, page backdrop, large empty canvas areas |
| Secondary (30%) | #1d242c | Section cards, form surfaces, disclosures, review containers |

## Layout Contract

Preserve the existing viewer-first dashboard. The page order for Phase 4 is:

1. Hero
2. Viewer + PTZ cluster
3. Mode switcher
4. Settings panel
5. Diagnostics disclosure

The settings surface sits below transport controls so live view remains primary, while diagnostics stay the last and most secondary surface.

Settings panel rules:

| Area | Contract |
|------|----------|
| Panel shell | Use the same card language as `.mode-switcher`, `.diagnostics-disclosure`, and `.ptz-panel`: 24px padding, 12px radius, 1px low-contrast border, subtle top-light gradient |
| Section layout | Stack section cards vertically with 24px gaps; do not use accordion nesting |
| Section heading | Reuse the current two-line pattern: uppercase support label in accent, then 20px heading |
| Desktop width | Match the current `viewer-dashboard` width and align edges with existing cards |
| Mobile behavior | Collapse every section to a single column; keep button rows wrapping rather than shrinking below 44px height |

Source: existing `App.tsx` shell composition and `web/src/styles.css`.

## Section Inventory

Use this exact order and editability split:

| Order | Section | Mode | Fields / Content | Notes |
|-------|---------|------|------------------|-------|
| 1 | Time & Sync | editable | `Ntp.enable`, `Ntp.server`, `Ntp.port`, `Ntp.interval`, `Time.timeZone`, `Time.timeFmt`, `Time.hourFmt`, `Dst.*` | Current clock values are inspect-only |
| 2 | Display Overlay | editable | `osdChannel.enable`, `osdChannel.name`, `osdChannel.pos`, `osdTime.enable`, `osdTime.pos` | Emphasize overlay clarity, not branding customization |
| 3 | Basic Image Tuning | editable | `bright`, `contrast`, `hue`, `saturation`, `sharpen` | Present as bounded tuning controls with explicit defaults/current values |
| 4 | Stream Profile | editable with warning | `subStream.size`, `subStream.frameRate`, `subStream.bitRate`, `subStream.profile`; main stream summary read-only | Show a warning that playback may briefly reset after apply |
| 5 | Camera Tuning | read-only | `Isp.*` and other safe inspect-only camera values | Label clearly as inspect-only and omit any edit affordance |

Do not show editable controls for network, storage, alarm, or unsupported camera categories in Phase 4.

Source: locked decisions D-01, D-02, D-09, D-10 and Phase 4 research field matrix.

## Field Patterns

Use these control patterns consistently:

| Pattern | Contract |
|---------|----------|
| Text inputs | Filled dark input on the secondary surface, 12px radius, 16px horizontal padding, 44px minimum height |
| Selects | Match text-input shape and height; use accent only on focus and selected option state |
| Boolean toggles | Use explicit labeled segmented buttons or checkbox rows, not switch-only affordances with hidden meaning |
| Numeric tuning | Pair each image field with a labeled slider plus a visible numeric value; do not hide the current value inside tooltip-only UI |
| Read-only rows | Two-column label/value rows with muted labels and high-contrast values |
| Section badges | Small inline status pill for `Draft`, `Review`, `Applying`, `Verified`, `Read only` |

Do not introduce icon-only actions in this phase. All section actions must use text labels.

Source: current text-first control language in `ModeSwitcher`, `PtzPanel`, and `DiagnosticsDisclosure`; new form controls defined here by default.

## Interaction Contract

Each editable section card owns its own state. There is no global save bar and no cross-section draft sharing.

Section state machine:

| State | UI Treatment | Required Actions |
|-------|--------------|------------------|
| Read | Current values only, no emphasis badge | `Edit` |
| Editing | Form controls visible, draft badge shown if values differ | `Cancel`, `Review Changes` |
| Review | Inline review callout inside the same card with before/after rows | `Back to Editing`, `Apply Settings` |
| Applying | Controls disabled, status badge changes to `Applying`, spinner optional but not required | none besides passive wait |
| Verified | Compact accent-tinted summary with verified badge and changed fields | `Edit Again`, `Refresh` optional |
| Error | Inline field errors and/or a section-level rejection panel in destructive color | `Fix and Review Again` |

Rules:

1. `Review Changes` stays disabled until at least one field differs from the current camera-backed value.
2. Review must happen inline in the same card. Do not use a modal.
3. `Apply Settings` sends only that section's payload.
4. After apply succeeds, re-read the camera and replace the review callout with a verified before/after summary.
5. Keep user draft values intact when the camera rejects a write.

Source: locked decisions D-03 through D-08 and research Pattern 4.

## Review And Verification Treatment

Use compact change summaries instead of raw payload dumps.

| Element | Contract |
|---------|----------|
| Review callout | Secondary-surface block with accent border and 16px internal spacing |
| Change row format | `Field label` + `Before -> After` on one line when short; stack on two lines when values wrap |
| Verified summary | Replace review callout after success with `Verified against camera` label and the confirmed changed rows |
| Stream warning | Add a muted amber note above `Apply Settings`: `Applying stream changes may briefly reset live playback.` |
| No-op result | If the verified re-read matches the original value, show `No camera change detected` as a section-level warning instead of a success state |

The verified summary is part of the section card, not a toast and not a history log.

Source: locked decisions D-05 and D-06; research on verified re-read requirement.

## Error And Empty States

Use inline specificity first, section-level fallback second.

| Situation | Contract |
|-----------|----------|
| Field validation failure | Show the message directly under the input in destructive color; keep the section in `Editing` state |
| Camera/API rejection | Show a section-level panel directly above section actions with the camera-facing reason translated into plain language when possible |
| Read-only unsupported section | Show values normally with a `Read only` badge and no disabled fake buttons |
| Section unavailable | Render the empty-state heading/body inside the section slot; do not remove surrounding panel spacing |
| Bootstrap loading | Use quiet loading copy inside each pending card area, not a full-screen blocker |

Copy tone rules:

1. State the problem directly.
2. Tell the user what to do next.
3. Avoid protocol jargon unless it helps recover from the issue.

Source: requirements `CONF-01` through `CONF-04`, current PTZ and viewer status copy patterns.

## Accessibility Contract

| Area | Contract |
|------|----------|
| Live status | Each editable section has one `role="status"` / `aria-live="polite"` region for review, verified, and error outcomes |
| Focus handling | After `Apply Settings` resolves, move focus to the section heading or status region so the result is announced immediately |
| Touch target | Every interactive settings control is at least 44px tall |
| Labels | Every field uses a visible text label; helper copy is not a replacement for labels |
| Read-only values | Use semantic text or definition-list style groupings, not disabled form controls, for inspect-only content |

Source: current live-region pattern in `ViewerStatusOverlay` and `PtzPanel`; defaults for new form controls.

## Motion And Tone

Keep the existing restrained motion language:

| Property | Contract |
|----------|----------|
| Hover/focus transitions | 150ms to 200ms ease |
| State color transitions | 200ms ease |
| Press feedback | 1px downward translate allowed for buttons only |
| Panel animation | None required beyond subtle opacity/height reveal for inline review content |

Do not add celebratory animation, floating toasts, or slide-over panels. This phase should feel careful and trustworthy, not fast or playful.

Source: current button and panel transitions in `web/src/styles.css`; Phase 4 trust-first product intent.
