## Layout Contract

- Desktop uses a viewer-first composition: a single dominant live-view frame at the top, transport and quality controls directly beneath it, and diagnostics tucked into a secondary panel below or to the right.
- Mobile keeps the same order in one column: viewer frame, status and retry row, mode switcher, diagnostics disclosure.
- The live-view frame owns the visual focus and should occupy at least 70% of the initial viewport height on desktop before any lower panels scroll into view.
- Viewer aspect ratio defaults to 16:9. When the incoming stream ratio differs, preserve the media and letterbox inside the frame rather than cropping camera output.
- PTZ, settings, and broader dashboard chrome stay out of Phase 2. Reserve lower-page space with neutral panels rather than inventing future controls now.

## Component Inventory

- `LiveViewerFrame`: media surface, poster/snapshot fallback, and in-frame state overlays.
- `ViewerStatusOverlay`: connecting, live, reconnecting, failed states presented inside the frame, not in a detached toast.
- `ModeSwitcher`: segmented control that separates transport from quality in labels, for example `WebRTC Main`, `HLS Sub`, `Snapshot`.
- `RetryButton`: primary CTA shown only in failed state and secondary in reconnecting state.
- `DiagnosticsDisclosure`: collapsed by default; exposes transport reason, current mode, and next fallback without cluttering the main viewer.
- `StatusDot`: compact semantic indicator paired with plain-language text.

## Interaction Contract

- Auto-start the preferred mode on page load. Do not gate initial playback behind a splash screen or click-to-play prompt. Source: CONTEXT D-03 and D-05.
- Startup order is fixed: `WebRTC Main -> WebRTC Sub -> HLS Sub -> Snapshot Main`. Source: RESEARCH recommended startup order.
- Manual mode switching is always available after startup if the capability matrix exposes another enabled mode. Source: CONTEXT D-04.
- User-selected mode remains active until it fails definitively. Do not immediately yank the user back to auto mode after a successful manual switch.
- Reconnect behavior uses visible backoff with attempts at 1s, 2s, and 5s before degrading to the next fallback mode. After the fallback ladder is exhausted, land in `failed`. Source: CONTEXT D-06, RESEARCH discretion area resolved here.
- Overlay states are exclusive and named exactly: `Connecting`, `Live`, `Reconnecting`, `Live View Failed`. Source: CONTEXT D-07.
- Each non-live state includes a short reason line. Keep it under 90 characters and explain the problem in plain language, for example `WebRTC handshake timed out` or `This stream codec is not browser-safe`.
- Never show an indefinite spinner by itself. Every spinner must be paired with a state label and current mode.
- Snapshot fallback uses the last successful still immediately, then low-frequency refresh at 1 fps while snapshot mode is active. Source: RESEARCH open question resolved here in favor of a light live-feel fallback.
- Diagnostics stay secondary. The default viewer shows human-readable reasons only; raw transport details live behind the disclosure panel. Source: CONTEXT discretion and existing debug-capture pattern.

## Visual Contract

- The viewer frame uses a high-contrast dark shell so status overlays remain legible against unpredictable camera footage.
- Surfaces are squared-off with a subtle 12px radius. Avoid glassmorphism, floating gradients, or entertainment-style neon treatment.
- Status overlays use a dense top-left block for state and reason, with actions aligned below. Do not center large modal cards over the video unless the viewer is fully failed.
- Motion is restrained: 150ms fade for overlay changes, 200ms background-color transition for mode selection, no bouncing or pulsing effects beyond a subtle reconnect dot.
- Focus states must be visible on every interactive control with a 2px accent outline that sits outside the component boundary.

## Accessibility Contract

- All viewer actions must be keyboard reachable in DOM order: retry, mode switcher, diagnostics disclosure.
- Status changes must be announced through a polite live region that mirrors the visible overlay label and reason text.
- Body text contrast must meet at least 4.5:1 against dominant and secondary surfaces; large heading text must meet at least 3:1.
- Do not encode state by color alone. Pair every state color with a label and, where space allows, an icon.
