## Phase Context Summary

Phase 7 is a gap-closure phase for real-camera E2E verification. The CONTEXT.md defines:

**5 Test Procedures:**
1. Viewer Connect - Start app, open browser, expect Live state within 10s
2. Retry Live View - Disconnect manually, click retry, expect recovery
3. PTZ Hold/Release - Hold pan/tilt, release, expect "Motion stopped"
4. Settings Apply - Edit field, click Apply, expect "Verified" after reread
5. Diagnostics - Trigger failure, open diagnostics, expect short reason (no credentials)

**Dependencies:**
- Phase 6 verified Phase 1 requirements (CONN-01 to COMP-03)
- Phase 6 confirmed capability snapshot wiring to Phase 2/3/4
- All phases verified upstream through Phase 6

**Failure Context Requirements:**
- Capture: timestamp, operation name, short reason, correlation refs
- DO NOT capture: credentials, tokens, passwords, raw RTSP URLs, camera auth tokens
</context>

| App → Camera | Internal LAN only, authenticated |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Information Disclosure | Diagnostics | mitigate | CONTEXT.md requires: DO NOT capture credentials/tokens/passwords in debug artifacts |
| T-07-02 | Denial of Service | Live View Recovery | accept | Real-camera networking may have transient failures on LAN |
| T-07-03 | Tampering | Settings Apply | mitigate | Settings are gated by capability snapshot, read-validate-write verification loop |

## Security Notes

- All credentials remain in `.local/camera.config.json` - not exposed to browser
- Diagnostics must not leak tokens, passwords, or raw RTSP URLs
- Settings writes go through verification loop (camera reread to confirm)
</threat_model>
